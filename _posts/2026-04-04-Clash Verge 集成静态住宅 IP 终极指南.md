# 📝 实战笔记：Clash Verge 集成静态住宅 IP 终极指南

### 1. 核心背景：为什么需要“链式代理”？
* **痛点**：直连海外住宅 IP（如 IPFLY 英国节点）极易出现 `Timeout`（超时）。
* **原因**：住宅 IP 端口（如 5001）特征明显且跨境链路长，易受运营商拦截或 GFW 干扰。
* **方案**：**链式代理（Proxy Chain）**。即：`南京本地` -> `机场高速节点 (入口)` -> `静态住宅 IP (出口)` -> `Claude API`。

### 2. 关键配置：Clash Verge 可视化操作
在 Clash Verge 中，通过“代理链配置”界面可以直观完成：

1.  **入口节点 (Input)**：选择一个延迟低、带宽大的机场节点（如 JMS 的 vmess 节点）。它负责“破墙”和加速。
2.  **出口节点 (Output)**：选择你的 **IPFLY 静态住宅 IP**（Socks5 协议）。它负责提供真实的英国/美国 ISP 身份。
3.  **结果**：最终延迟 = 入口延迟 + 出口延迟。实测约 **400ms-500ms** 为正常范围，足以支撑 `claudecode` 的交互。

### 3. 环境保障：虚拟网卡模式 (TUN)
对于开发者（尤其是使用命令行工具如 `claudecode`, `git`, `ssh`）来说，这是**必须开启**的选项。

* **作用**：在系统层面拦截所有网络流量，强制导入代理链。
* **优点**：无需在终端手动 `export http_proxy`，解决 Node.js 运行环境不走系统代理的问题。
* **操作**：在 Clash Verge 首页点击 **“虚拟网卡模式”**（TUN Mode），确保开关为蓝色并显示“已启用”。

### 4. 配置文件 (YAML) 核心逻辑
如果你偏好手动维护配置文件，核心代码如下：

```yaml
proxies:
  - {name: "JMS-Entry", type: vmess, server: ..., port: ...} # 机场节点
  - {name: "IPFLY-Exit", type: socks5, server: ..., port: 5001, dialer-proxy: "JMS-Entry"} # 住宅IP挂载在入口上

proxy-groups:
  - name: "Claude-Relay"
    type: relay
    proxies:
      - "JMS-Entry"
      - "IPFLY-Exit"
```

### 5. 开发者避坑指南
* **协议检查**：若 Socks5 超时，尝试切换为 HTTP 协议。
* **身份验证**：确保 `ipinfo.io` 返回的是住宅 IP（如 `149.88.212.87`），且国家为英国。
* **账号安全**：固定使用同一个“出口”住宅 IP 访问 Claude，避免因 IP 频繁变动触发风控封号。
* **6.30 目标**：这套网络环境是实现“AI 辅助编码完整作业流”的物理底座，建议保持配置不动，避免开发中断。

---
**💡 总结**：**“机场跳板 + 住宅落地 + TUN 模式”** 是目前兼顾速度、稳定性与账号安全的最优解。
